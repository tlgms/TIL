## 2. 성능 단위: FLOPs vs FLOP/s

- **FLOPs** (FLoating-point OPerations) = 부동소수점 **연산 개수** → *일의 양*

- **FLOP/s** (FLOP per second) = 초당 연산 수 → *속도 / 성능*

```plain text
실행시간 = FLOPs ÷ FLOP/s
```

- 알고리즘 개선 → 분자(FLOPs)를 줄임

- 구현 개선 → 분모(FLOP/s)를 키움

- **곱하고 더하는 걸 대량 반복** → 구현 최적화 연습에 딱 좋은 대상

## 4. 메모리 계층 ⭐ (가장 중요)

CPU는 계산은 빠르지만 **데이터 가져오기가 느림.** 그래서 계층이 존재:

```plain text
레지스터  →  캐시(L1/L2/L3)  →  메인 메모리
(빠름/작음)                    (느림/큼)
```

- **Memory → 레지스터**: 실제 연산은 레지스터 값으로만 수행. 자주 쓰는 값을 레지스터에 붙잡아 두면 메모리 왕복이 줄어듦.

- **캐시 라인 = 64바이트** *(필기의 "64비트"는 오타 → 바이트가 정답)*: 캐시는 한 번에 64바이트를 **통째로** 가져옴. 데이터를 **연속(순서대로)** 접근하면 그 라인을 알뜰하게 다 씀.

- **캐시 프리페칭 (Prefetching)**: "다음에 쓸 데이터"를 CPU가 **미리** 캐시로 당겨놓음 → 대기 시간 은닉.

- **캐시 최적화**: 위 성질을 노려 데이터 접근 패턴을 설계.

---

## 5. 루프 최적화

- **루프 순서 바꾸기**: 접근 순서를 메모리 저장 순서에 맞추면 캐시 적중률 급상승. (행렬 곱 `ijk` vs `ikj`가 대표 예시)

- **루프 불변 코드 (Loop-invariant code motion)**: 루프 안에서 값이 안 변하는 계산을 **루프 밖으로** 빼서 반복 계산 제거.

- **루프 언롤링 (Unrolling)**: 반복문을 펼쳐 반복 관리 비용(조건 검사·점프)을 줄이는 기법.
  - ⚠️ **요즘은 애매**: 현대 CPU가 **비순차 실행(Out-of-order execution)**으로 이미 알아서 최적화 → 손으로 언롤링해도 이득이 적거나 코드만 지저분해질 수 있음.

---

## 6. SIMD (Single Instruction, Multiple Data)

- 명령 하나로 **여러 데이터를 동시에** 처리 (예: float 8개를 한 번의 곱셈 명령으로)

- "같은 연산을 데이터마다 반복"하는 합성곱 같은 작업에 완벽하게 맞음

---

## 7. 멀티스레딩 / 멀티코어

- 앞의 내용이 **코어 1개**를 잘 쓰는 법이라면, 이건 **코어 여러 개**에 일을 분배

- 스레드를 여러 개 만들어 각 코어가 데이터의 다른 부분을 동시에 계산 → 이론상 코어 수만큼 가속

---

## 한 줄 요약

> 계산량(FLOPs)은 그대로 두고,
① 데이터를 **캐시/레지스터**에 잘 태우고
② **SIMD**로 한 번에 여러 개 처리하고
③ **여러 코어**로 나눠서
→ **FLOP/s(속도)를 끌어올리는 것**이 이 단원의 전부.

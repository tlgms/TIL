# 쿠키& 세션

### http의 특성상 로그인의 정보등이 저장되지 않는다. 이러한 단점을 보안하기 위하여 쿠키와 세션은 웹 통신 정보를 저장해준다.

## 세션 넌 뭐하는 애야?
> 이 글을 정리하게 된 이유는 세션에 대한 개념을 다지고, 세션을 검색하면 JSP로 구현되어있는 구글의 예제 코드들에 빡쳐서 정리하게 되었다.
내가 재학중인 학교에서는 로그인, 회원가입을 구현할 때 JWT로 구현을 많이 한다. (사실 모든 프로젝트에 JWT를 쓴 것 같다😅) 그래서 그런지 세션과 쿠키에 대해 개념만 알거나 개념조차 모르는 경우가 많다. 나도 그 중 하나였다… 
새로 하는 프로젝트에서 로그인/회원가입을 담당하게 되었는데 OAuth와 세션을 사용하게 되어서 세션에 대해 공부하던 중 위의 이유로 세션에 대해서 정리하게 되었다.
## 그래서 **세션**이 뭐지?
HTTP는 서버가 클라이언트의 상태를 저장하지 않는(무상태성) 특징을 가지고 있기때문에
> [client] 저 로그인 할께요! 제 이름은 OOO이고, 비밀번호는 1234로 할께요.
  [server] 넵
[client] 나 OOO인데 비밀번호를 까먹었어 알려줘!
[server] 누구죠?
위 상황처럼 클라이언트에서는 분명 로그인을 했지만, 서버는 클라이언트를 알아볼 수 없는 상태가 발생하게 된다.
이를 보완하기 위해 세션을 사용한다. 일단 세션을 알기 위해서 먼저 쿠키부터 무었인지 알아볼까한다.
### Cookie(쿠키)란?
쿠키는 웹 브라우저 로컬에 저장되는 키와 값이 들어있는 작은 데이터(텍스트) 파일이다.
- 쿠키는 클라이언트에 저장된다.
- 웹 브라우저를 종료하면 쿠키도 없어진다. → 만료시간을 정하면 만료일 전까지는 사라지지 않는다.
- 쿠키 한개에 4KB까지 최대 300개까지 저장할 수 있다. 하나의 도메인당 30개 저장 가능
- 웹 브라우저에 해당 서버의 쿠키정보가 있다면 HTTP 해더에 쿠키를 담아서 보낸다.
쿠키의 목적
1. 세션 관리 : 로그인, 장바구니, 게임스코어 등의 정보를 관리
1. 개인화 : 사용자만의 테마 등의 세팅
1. 트레킹 : 사용자의 행동을 기록, 분석
쿠키의 구성 요소
- Name : 쿠키의 이름
- Value : 쿠키의 저장된 값
- Expires : 쿠키의 만료일 (만료일 포함시 영구적 쿠키, 지정된 만료일이 되면 쿠키 삭제)
- Domain : 쿠키가 사용하는 도메인 
(현재 도메인과 일치하지 않으면, 타사 쿠키로 간주하여 브라우저에서 거부 → 다른 도메인의 쿠키 사용 X)
- Path : 쿠키가 반환할 경로를 결정
- Secure : 보안 연결 설정
- HttpOnly : HTTP 외에 다른 통신 사용 가능 설정
쿠키의 종류
### Session(세션)이란?
이제 드디어 세션에 대해서 알아보도록하겠다.
세션은 방문자가 웹 서버에 접속해 있는 상태(연결 시작 ↔ 연결 해제까지의 상태)의 기간이라고 생각하면 된다.
> 서버에 세션에 대한 정보를 저장한 후 클라이언트에게 세션 쿠키를 주어 서버가 클라이언트를 식별할 수 있게 되는 방식 자체를 의미하는 경우가 많다.
- 용량 제한이 없다.
- 쿠키를 사용하여 Session ID 값을 클라이언트에게 보낸다. 
(이때 사용되는 세션 ID 값은 고유한 추정 불가능한 랜덤값이어야한다 → UUID등)
- 브라우저를 닫거나 서버에서 세션을 삭제 했을 시에만 삭제 된다.
- 웹서버에 저장되는 쿠키이다. (= 세션 쿠키)
  - 웹 서버에 저장되기 때문에 서버의 자원을 사용한다.
